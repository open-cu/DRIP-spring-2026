Блок batch-retry: пакетная обработка и повторные попытки

Бизнес-смысл: батчи помогают снижать нагрузку на БД при массовых событиях, а retry защищает процесс от временных ошибок интеграций.

Страница обновляет метрики автоматически каждые 2 секунды, чтобы наглядно показать динамику без F5.

Блок 1. Batch Processing

Сценарий массовой обработки: отправляем пачку сообщений и наблюдаем, как сервис сохраняет их группой.

Параметры batching:
producerSize = 20 — сколько сообщений копим перед отправкой одним batch.
producerTimeoutMs = 500 — через сколько ms отправим неполный batch.
producerBufferLimit = 1048576 — лимит буфера batch у producer.
consumerSize = 20 — максимальный размер пачки, которую получает listener.
consumerReceiveTimeoutMs = 1000 — время ожидания добора batch на consumer.

Блок 2. Retry

Сценарий устойчивости: отправляем заказ и смотрим, как работают повторные попытки при ошибке.

Параметры @Retryable:
maxAttempts = 5
backoff = @Backoff(delay = 1000, multiplier = 2.0)
Паузы растут геометрически: 1s, 2s, 4s, 8s ... до последней попытки.
