Блок kafka-delayed-processing: отложенный заказ

Учебный сценарий: заказ сначала получает статус ОЖИДАЕТ, событие попадает в delay topic, а обработка запускается после времени processAt.

Consumer не делает sleep: если время еще не пришло, Spring Kafka ставит partition на паузу и читает сообщение снова позже.

Если обработка после задержки падает, Spring Kafka проводит событие через retry topic и только потом отправляет его в DLT.

Сценарий

Пользователь создает отложенный заказ и выбирает задержку: 5, 30 или 300 секунд. Заказ сразу получает статус ОЖИДАЕТ, а событие сначала попадает в request topic (канал запроса на задержку) с заголовком processAt.

Если consumer читает событие раньше срока, он не коммитит offset и вызывает backoff manager. Spring Kafka временно ставит partition на паузу, затем снова отдает это же сообщение, consumer отправляет событие в process topic, и второй listener переводит заказ в статус ОБРАБОТАН.

  • Компромисс 1: listener временно не читает следующие сообщения, поэтому возможна очередь за первым отложенным событием.
  • Компромисс 2: нужен ручной commit. Нельзя фиксировать offset, пока событие еще только ждет своего времени.
  • Компромисс 3: delay живет в consumer-логике, потому что Kafka broker не дает отдельной кнопки delayed delivery.
  • TaskScheduler здесь инфраструктурный: он нужен Spring Kafka, чтобы снять partition с паузы, а не чтобы запускать бизнес-обработку.
  • Технический retry начинается только после process topic: это не бизнес-задержка, а повтор при ошибке обработки.
  • RetryTopicConfiguration отправляет событие в retry topic, ждет интервал и возвращает его в обработку.
  • DLT (topic для проблемных сообщений) нужен, когда событие нельзя обработать после всех попыток.

Действие для демонстрации

Для демонстрации retry и DLT введите описание со словом fail.
